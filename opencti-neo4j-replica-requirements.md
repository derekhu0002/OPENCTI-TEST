# OpenCTI 远端 Neo4j 副本与独立查询 API 需求说明

## 1. 背景

当前 AI4X-Platform 在消费 OpenCTI 数据时，采用的是“Cypher 查询语义 -> OpenCTI GraphQL 查询”的翻译方式。该方案在基础检索场景下可工作，但在以下两类场景中存在明显限制：

1. 查询结果存在缺失或不完整问题。
2. 部分 Cypher 高级查询能力无法等价表达或无法执行。

这些问题使得 OpenCTI 数据难以稳定支撑复杂图查询、路径分析、组合过滤、多跳遍历及图分析类能力。

为解决上述问题，计划评估在 OpenCTI 所在远端环境中建设一个面向查询优化的 Neo4j 只读副本，并提供一个独立的查询 API，专门承接高级图查询场景。

本需求文档用于提交给 OpenCTI 项目方评估其可行性、实施方式、边界条件及所需配合项。

## 2. 目标

本需求的目标如下：

1. 在 OpenCTI 所在远端环境中提供一份面向查询优化的 Neo4j 图副本。
2. 该副本中的数据来源于 OpenCTI 主系统，作为只读分析副本使用，而非新的主数据源。
3. 在副本旁提供独立查询 API，用于执行高级图查询、关系遍历、路径分析及复杂过滤。
4. 保持 OpenCTI 作为权威主源，现有对象写入、治理、权限和主数据管理能力仍以 OpenCTI 为准。
5. 为 AI4X-Platform 提供稳定、可预期、可扩展的高级查询能力，避免继续依赖 Cypher 到 GraphQL 的不完全翻译。

## 3. 业务问题

当前方案主要存在以下业务和技术问题：

### 3.1 查询翻译不完整

- 一部分 Cypher 条件无法完整翻译为 OpenCTI GraphQL 查询。
- 一部分查询虽然可以翻译，但会退化为有限扫描加本地过滤，不能保证全量语义完整性。
- 当查询涉及复杂过滤、多条件组合或图遍历时，返回结果可能出现漏数或不稳定。

### 3.2 高级图查询能力受限

- 多跳关系遍历能力有限。
- 路径查询和子图提取能力不足。
- 复杂布尔组合、聚合和图模式匹配能力受限。
- 一些 Cypher 高级能力无法在 OpenCTI GraphQL 侧等价实现。

### 3.3 架构扩展成本高

- 继续扩展“Cypher -> GraphQL”翻译器的收益递减。
- 翻译器需要持续维护语法子集、字段映射和退化策略，复杂度高。
- 即使持续增强翻译器，仍无法从根本上获得等价的图数据库查询语义。

## 4. 需求概述

建议由 OpenCTI 项目方评估以下能力是否可支持：

1. 在 OpenCTI 所在远端环境中部署一个 Neo4j 图数据库实例，作为 OpenCTI 数据的只读查询副本。
2. 提供从 OpenCTI 到该 Neo4j 副本的数据同步机制。
3. 在副本所在环境旁部署一个独立查询 API，用于对 Neo4j 副本执行高级图查询。
4. 提供必要的数据新鲜度、同步状态和失败处理机制，以便调用方判断副本可用性和时效性。

## 5. 总体需求

### 5.1 副本定位

- Neo4j 副本为只读分析副本。
- OpenCTI 仍为权威主数据源。
- 不要求在副本侧承接写入、编辑、对象治理或主数据管理能力。
- 副本服务于高级查询和图分析，不替代 OpenCTI 原生对象查询和管理能力。

### 5.2 部署位置

- Neo4j 副本应部署在 OpenCTI 所在远端环境中，或至少部署在与 OpenCTI 具备低延迟、稳定网络连通的同一环境内。
- 独立查询 API 应尽量靠近 Neo4j 副本部署，以减少大结果集跨网络传输成本。
- 不建议将 Neo4j 直接嵌入 OpenCTI 数据库内部，而应作为独立但邻近的图查询组件。

### 5.3 查询服务定位

- 独立查询 API 应为只读服务。
- API 主要承接高级图查询场景，不要求替代 OpenCTI 全部查询接口。
- API 应支持受控方式执行图查询，避免无限制、高风险的任意查询直通。

## 6. 数据同步需求

### 6.1 同步目标

需要将 OpenCTI 中适用于图分析的对象和关系同步到 Neo4j 副本，以支持：

- 高级关系遍历
- 路径分析
- 子图提取
- 复杂过滤与组合查询
- 图聚合与图分析

### 6.2 同步范围

请 OpenCTI 项目方评估以下同步范围是否可支持：

1. 近期变更对象增量同步。
2. 相关关系同步。
3. 为保证图完整性，对近期变更对象的 1 到 2 跳邻居进行补齐同步。
4. 对关键实体类型提供全量或准全量常驻同步能力。

关键实体类型建议至少包括：

- Threat-Actor
- Malware
- Indicator
- Campaign
- Report
- Attack-Pattern
- Vulnerability
- Infrastructure
- Intrusion-Set
- Tool

### 6.3 同步方式

请评估以下方案的可行性：

1. 基于更新时间字段进行增量拉取同步。
2. 使用 watermark 记录上次成功同步位置。
3. 周期性进行全量校准或抽样对账，补齐增量遗漏。
4. 对对象、关系分别进行幂等 upsert。

### 6.4 删除与失效处理

请 OpenCTI 项目方评估以下能力是否可提供：

- 删除对象同步到副本的策略。
- 关系删除同步策略。
- 软删除、撤销、失效标记或 tombstone 支持方式。
- 副本数据与主源不一致时的修复策略。

### 6.5 标识与幂等性

建议在副本中使用 OpenCTI 对象的稳定标识作为主键，例如：

- `standard_id` 作为稳定对象标识。
- 保留 OpenCTI 内部对象 ID 作为外部追踪字段。
- 关系使用“关系类型 + source_id + target_id”进行幂等定位。

请 OpenCTI 项目方确认：

- 哪个字段最适合作为稳定主键。
- 是否存在适用于跨同步任务的关系唯一标识。

## 7. 独立查询 API 需求

### 7.1 API 定位

独立查询 API 面向以下场景：

- 复杂图模式查询
- 多跳遍历
- 路径查找
- 子图抽取
- 受控 Cypher 查询
- 图统计与图分析

### 7.2 API 能力要求

建议独立查询 API 至少提供以下能力：

1. 按实体类型、时间范围、标签、关系类型进行图查询。
2. 支持 1 跳、2 跳及可配置跳数的关系扩展。
3. 支持指定起点或终点的路径查询。
4. 支持按对象属性和关系属性进行组合过滤。
5. 支持子图返回。
6. 支持可审计、受限的 Cypher 查询能力。
7. 返回副本数据新鲜度信息，例如最近同步时间、同步延迟、数据版本号等。

### 7.3 安全与控制

API 不应默认开放无限制的任意 Cypher。建议具备以下控制能力：

- 鉴权认证
- 审计日志
- 查询超时控制
- 结果条数限制
- 资源配额控制
- 允许查询的 schema 或模式白名单

### 7.4 响应元数据

建议 API 响应中包含以下元数据：

- `backend=neo4j-replica`
- `freshness_ts`
- `staleness_seconds`
- `sync_status`
- `query_timeout_applied`
- `result_truncated`

## 8. 非功能需求

### 8.1 性能

- 支持高级图查询的响应时间显著优于当前 GraphQL 翻译方案。
- 对常见 1 到 2 跳遍历、关系分析和子图提取请求，期望在可接受时间内返回。
- 支持并发查询，不明显影响 OpenCTI 主系统稳定性。

### 8.2 稳定性

- 副本同步失败不应影响 OpenCTI 主系统正常运行。
- 查询 API 失败不应破坏主系统数据。
- 同步与查询服务应具备独立监控和故障恢复机制。

### 8.3 可观测性

建议具备以下观测能力：

- 最近同步成功时间
- 当前同步延迟
- 同步失败次数
- 对账结果
- API 查询耗时
- API 错误率
- 查询超时和限流情况

### 8.4 可扩展性

- 支持未来扩展更多实体类型和关系类型。
- 支持未来扩展更多图分析 API。
- 支持未来由 AI4X-Platform 或其他消费方接入。

## 9. 权限与数据边界

请 OpenCTI 项目方重点评估以下问题：

1. OpenCTI 当前的数据访问控制模型是否需要同步到副本查询 API。
2. 如果 OpenCTI 存在基于用户、组织、租户或数据域的访问边界，副本 API 应如何继承或映射该权限模型。
3. 是否允许副本 API 暴露比 OpenCTI 原生接口更宽的数据可见性。

本需求建议：

- 副本 API 不应绕过 OpenCTI 既有的数据访问边界。
- 若短期内无法同步细粒度权限，至少需要具备服务级鉴权和受控访问能力。

## 10. 期望评估问题

希望 OpenCTI 项目方评估并反馈以下问题：

1. 是否支持在 OpenCTI 远端环境中部署独立 Neo4j 查询副本。
2. 是否有现成或推荐的数据导出、事件通知、增量同步或 CDC 机制可复用。
3. 哪些对象与关系字段适合作为副本主键和幂等同步依据。
4. 删除、撤销、失效对象是否可被稳定同步。
5. 是否建议同步全量图，还是建议同步热子图。
6. 独立查询 API 是否由 OpenCTI 项目方提供、共建，或由我方在远端环境中自建更合适。
7. 若提供独立查询 API，推荐的鉴权、审计与限流方式是什么。
8. 该方案对 OpenCTI 主系统性能、资源占用和运维复杂度的影响如何。
9. 该方案实施的风险点、前置条件和推荐阶段划分是什么。

## 11. 建议的范围边界

为便于评估，建议本期需求边界如下：

### 11.1 本期纳入

- Neo4j 只读副本方案评估
- OpenCTI 到 Neo4j 的同步方案评估
- 独立查询 API 方案评估
- 鉴权、审计、同步状态和新鲜度能力评估
- 面向高级图查询场景的可行性评估

### 11.2 本期不要求

- 由 Neo4j 副本承接 OpenCTI 主写入能力
- 替代 OpenCTI 的全部对象管理能力
- 完全复制 OpenCTI 全部功能接口
- 在本期内解决所有权限细粒度继承问题
- 在本期内实现任意无限制通用 Cypher 开放

## 12. 预期收益

如果该方案可行，预期收益包括：

1. 显著降低当前 Cypher 到 GraphQL 翻译的不完整性问题。
2. 为高级图查询提供稳定执行底座。
3. 降低复杂查询对 OpenCTI 主系统的直接压力。
4. 为 AI4X-Platform 后续图分析、关联研判、路径分析和知识图谱能力演进提供基础设施。

## 13. 希望 OpenCTI 项目方的反馈形式

希望对方评估后至少反馈以下内容：

1. 可行或不可行结论。
2. 推荐实现方式。
3. 推荐同步机制。
4. 需要我方配合的接口、部署、权限或网络条件。
5. 风险、限制和替代方案。
6. 推荐的分阶段实施路径。

## 14. 一句话需求总结

希望在 OpenCTI 所在远端环境中建设一个面向高级图查询的 Neo4j 只读副本及独立查询 API，以解决当前基于 GraphQL 翻译方式存在的查询不完整和高级 Cypher 能力缺失问题，同时保持 OpenCTI 作为权威主数据源。
